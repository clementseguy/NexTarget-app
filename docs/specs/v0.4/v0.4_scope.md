## Ajouts Techniques

1. Intégration SonarCloud
   - Utiliser SonarCloud connecté au dépôt GitHub.
   - Ajouter un job GitHub Actions :
       - Analyse automatique à chaque push sur `dev`
       - Analyse obligatoire sur chaque MR vers `main`
   - Critère d’acceptation : badge SonarCloud visible, note ≥ B, couverture tests ≥ 20 %.

2. Cahier de recette (tests manuels)
   - Générer automatiquement un document Markdown listant les fonctionnalités principales **existantes** de l’app.
   - Pour chaque fonctionnalité : nom, objectif, étapes de test, résultat attendu.
   - Sortie : `/docs/cahier_recette.md`
   - Objectif : permettre à un humain de tester l’app après refactoring (vérification absence de régression).

3. Refactoring sans régression
   - **✅ FAIT** : Couches UI, business et data séparées (aucun appel réseau direct dans les widgets)
   - **✅ FAIT** : Dashboard refactorisé avec architecture modulaire (DashboardService, composants réutilisables)
   - **✅ FAIT** : Suppression code mort (`lib/models.dart`, `lib/screens/exercises_screen.dart`)
   - **✅ FAIT** : Extraction `ExerciseFormScreen` (773L → 480L + 304L)
   - **✅ FAIT** : Modularisation `session_form.dart` (663L → 471L + 322L components)
   - **✅ FAIT** : Refactoring `planned_session_wizard.dart` (572L → 320L + 364L wizard_steps)
   - **✅ FAIT** : Modularisation `session_detail_screen.dart` (556L → 204L + 477L components)
   - **✅ FAIT** : Tous les fichiers désormais < 500 lignes (objectif architecture modulaire atteint)
   - **✅ FAIT** : Analyse statique complétée - aucun composant inutilisé détecté
   - **🟡 À FAIRE** : Tests unitaires pour composants extraits (priorité basse - app validée manuellement)

4. Authentification via NexTarget-server **EN COURS**
   - **✅ FAIT** : Backend `NexTarget-server` comme provider d'authentification centralisée
   - **✅ FAIT** : OAuth2 Google via flux personnalisé (HTTP GET → auth_url → navigateur externe → deep link callback)
   - **✅ FAIT** : Architecture mobile avec `app_links` + `url_launcher` + échange de tokens sécurisé
   - **✅ FAIT** : Stockage sécurisé JWT via `flutter_secure_storage`
   - **✅ FAIT** : Reconnexion automatique avec vérification token au démarrage
   - **✅ FAIT** : Navigation fluide post-authentification sans boucles de redirection
   - **✅ FAIT** : Logs sécurisés (aucun token visible en production)
   - **🟡 À FAIRE** : Auth Facebook
   - **🟡 À FAIRE** : Migration profils locaux vers comptes authentifiés (et vérification de l'absence de perte de données)

5. Déport des appels Mistral API vers NexTarget-server
   - Les specs API de NexTarget-Server seront communiquées
   - Le client mobile enverra :
       - l'action à exécuter (ie. à date, "analyse_session")
       - contexte (userId, Session informations)
   - Le serveur appelle Mistral API avec sa clé privée.
   - Le client ne stocke ni clé ni prompt.
   - Sécurité : appels HTTPS + token JWT utilisateur.

## Dashboard

Le refactoring du dashboard a généré des régressions, il a été décidé de supprimer l'existant et de redévelopper les widgets.
Les réalisations listées ici sont une conséquence de cette situation.

### D1 : Dashboard harmonisation et graphiques multi-courbes ✅ **FAIT**
- **Objectif** : Refactorisation du composant `EvolutionChart` pour supporter les graphiques multi-courbes avec dual-axis
- **Réalisations** :
  - Architecture générique pour 1 ou 2 courbes simultanées
  - Support dual-axis (pts à gauche, cm à droite) avec normalisation/dénormalisation automatique
  - Tooltips avancés avec informations contextuelles (date, index série)
  - Tests complets (16/16 passent)

### D2 : Graphiques évolution par méthode de prise ✅ **FAIT**
- **Objectif** : Ajouter des graphiques spécifiques aux méthodes de prise (1 main vs 2 mains)
- **Réalisations** :
  - "Scores et Groupement - 2 mains" (courbes amber/blue)
  - "Scores et Groupement - 1 main" (courbes orangeAccent/redAccent)
  - Service générique `generateHandMethodEvolutionData(HandMethod)` 
  - Filtrage intelligent par `HandMethod` avec méthode de compatibilité
  - Architecture DRY avec réutilisation complète du composant `EvolutionChart`

### D3 : Statistique "Prise dominante" ✅ **FAIT**
- **Objectif** : Remplacer "Catégorie dominante" par une métrique plus pertinente
- **Réalisations** :
  - Modèle `AdvancedStatsData` refactorisé (dominantHandMethod/dominantHandMethodPercentage)
  - Logique de calcul basée sur toutes les séries utilisateur avec pourcentages précis
  - Interface mise à jour : "1 main (XX.X%)" ou "2 mains (XX.X%)"
  - Icône appropriée (back_hand) et tests mis à jour

### Règles de sécurité

1. Feature : Règles de sécurité FFTir
   - Revoir le contenu textuel des règles dans le tableau de bord. Afficher précisément :
        1. Une arme doit toujours être considérée comme chargée et prête à tirer.
        2. Une arme ne doit jamais être dirigée vers quelqu'un.
        3. Avant de manipuler une arme, on doit toujours s'assurer qu'elle est mise en sécurité (contrôles visuel et physique)

   - Bouton "dupliquer" présent dans la vue Edition, à côté du bouton Enregistrer

   - **Prérequis** : Utilisation du composant générique `EvolutionChart` (déjà disponible ✅)
   - Données :
      - 2 courbes : Points + Groupement de chaque série de la session
      - Pas de tendance
      - Se limite aux données des séries de la session (ni plus, ni moins)
   - Placement : au-dessus de la section du coach
   - Apparence, titre et légende : conforme avec Dashboard
   - Conditions : (il y a au moins 3 séries)
